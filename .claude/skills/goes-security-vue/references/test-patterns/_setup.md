# Basic Setup — Use in Every Security Spec File

## Vitest config: `vitest.security.config.ts`

Create this at the repo root (next to `vite.config.ts`):

```typescript
import { defineConfig } from 'vitest/config'
import vue from '@vitejs/plugin-vue'
import path from 'node:path'

// The reporter lives under `.claude/skills/goes-security-vue/reporter/`.
// We alias it as `@security-reporter` so specs import cleanly.
const reporterPath = path.resolve(
  __dirname,
  '.claude/skills/goes-security-vue/reporter',
)

export default defineConfig({
  plugins: [vue()],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
      '@security-reporter': reporterPath,
    },
  },
  test: {
    environment: 'jsdom',
    include: ['tests/security/**/*.security-html.spec.ts'],
    reporters: [
      'default',
      [reporterPath + '/html-reporter.cjs', {
        outputPath: './reports/security/security-report.html',
      }],
    ],
  },
})
```

## Script `package.json`

```json
{
  "scripts": {
    "test:security": "vitest run --config vitest.security.config.ts"
  },
  "devDependencies": {
    "xlsx": "^0.18.5"
  }
}
```

The reporter produces BOTH artefacts on every run:

- `reports/security/security-report.html` (self-contained, opens in any browser)
- `reports/security/security-report.xlsx` (one sheet per bucket: Pasados / Desactivados / Migrados / No aplicables)

## Why `.cjs`?

Vue projects generated by `create-vue` / Vite ship `"type": "module"` in `package.json`. Under that mode Node treats `.js` files as ESM; Jest/Vitest custom reporters are CommonJS. The `.cjs` extension forces CJS parsing unconditionally.

## Basic spec shape

```typescript
import { describe, expect, it, vi } from 'vitest'
import { report } from '@security-reporter/metadata'

describe('[GOES Security FE] <component/module name>', () => {
  it('[Rxx] <assertion headline>', async () => {
    const t = report()
    t.epic('<area>')                 // e.g. Access Control, Session Management
    t.feature('<feature>')           // e.g. Router guard, Refresh coordinator
    t.story('<scenario>')            // e.g. Unauth visit redirects to /login
    t.severity('critical')           // blocker | critical | high | normal | minor
    t.tag('Pentest', 'OWASP-A01', 'GOES-R9')
    t.descriptionHtml('<p>Why this test exists, what attack it prevents.</p>')

    // ... Arrange / Act / Assert ...

    await t.flush()
  })
})
```

## IMPORTANT — adapt to the real project

All patterns in this folder are **reference examples** — they assume paths like `@/features/auth/stores/auth.store`. Before pasting a pattern into a real spec:

- Read the actual store / composable / service file path in the project
- Adjust imports (`@/features/...` vs `@/stores/...` vs whatever the project uses)
- Use the real method names (`login()` might be `signIn()`, `setSession()` might be `setToken()`)
- Match the real error copy (for example: the `authService` might return `"Invalid credentials"` instead of `"Credenciales invalidas"`)

Never copy a pattern verbatim without reading the source code first.
